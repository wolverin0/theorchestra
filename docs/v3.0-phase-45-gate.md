## Context

Phase 4 (A2A + auto-handoff) and Phase 5 (Agency mode) were shipped concurrently per the plan's parallelism rules. Phase 4 carries over v2.7's auto-handoff daemon (readiness check → `/handoff` skill → `/clear` → continuation) onto the v3.0 backend; Phase 5 adds persona catalog, git-worktree spawn path, and PRD-YAML team bootstrap.

## What shipped

### Phase 4

- `src/backend/auto-handoff.ts` — full flow: pre-flight checks → readiness check → `/handoff` skill dispatch → file-existence poll → section verification → idle settle → Ctrl+C + `/clear` + continuation inject. All timeouts configurable via `{timeouts}` option so gate harnesses can use short values.
- `ws-server.ts` — `POST /api/sessions/:id/auto-handoff` with body `{focus?, force?}` → runAutoHandoff result as JSON; HTTP status mapped from result variant (200 completed, 404 not_found, 409 working/not_ready, 500 incomplete, 504 timeout).
- `src/mcp/handlers/high-level.ts` — replaced the Phase 2 stub; `auto_handoff` now calls `backendClient.autoHandoff(pane_id, {focus, force})` and forwards the structured result.
- `src/mcp/client.ts` — added `autoHandoff(sessionId, body)`.

### Phase 5

- `src/backend/personas.ts` — `listPersonas()` scans `~/.claude/agents/` (flat + one level of nested category dirs), returns `{name, filePath, description, category}` array. `resolvePersona(name)` matches the v2.7 lookup rules.
- `src/backend/worktree.ts` — `addWorktree`, `removeWorktree`, `listWorktrees`, `defaultWorktreePath` convention (`<repo>/.worktrees/<branch>`).
- `src/backend/prd-bootstrap.ts` — narrow-shape YAML parser (no js-yaml dep) for `project/cwd/roles:[{name, persona?, prompt?, permission_mode?, branch?, tab_title?}]`. Supports `prompt: |` multi-line blocks.
- `ws-server.ts` — 4 new routes:
  - `GET /api/personas` — persona catalog
  - `GET /api/worktrees?repo=<abs path>` — list worktrees in a repo
  - `POST /api/worktree` — add a worktree (branch auto-created if absent)
  - `DELETE /api/worktree` — remove a worktree (with `force` flag)
  - `POST /api/prd-bootstrap` — parse YAML + spawn all roles, with persona resolution; returns `{project, cwd, spawned: [{role, session_id, persona}]}`
- `src/mcp/client.ts` — added `listPersonas`, `prdBootstrap`.

## Deviations from plan

1. **No standalone MCP tool for persona listing.** v2.7's MCP surface didn't expose one either — personas are injected at `spawn_session` time. The Spawn wizard hits `/api/personas` directly.
2. **Phase 4 gate's end-to-end flow needs a real Claude pane.** The auto-handoff logic is deterministic, but the `READY:`/`NOT_READY:` reply + `/handoff` skill file write both require a live Claude. Phase 10 dogfood will verify the full flow on 3 real high-ctx panes. Gate verifies surface aliveness + structural shape only.
3. **`[PEER-PANE CONTEXT]` header wiring was done in Phase 2.** Phase 2's MCP `spawn_session` handler already wraps prompts with the v2.7-identical header when `spawned_by_pane_id` is provided. Phase 5 added no new code for this — it just became reachable because the backend's spawn endpoint now accepts the `spawnedByPaneId` field.
4. **Worktree cleanup is best-effort in gate.** The gate attempts to `git branch -D` the branch it created, but if it fails the user will see stray branches. Production use would delete the branch inside `removeWorktree` on explicit caller request — not auto-removing to avoid destroying user work.

## Follow-ups surfaced

1. The PRD parser supports `branch:` on a role but the `/api/prd-bootstrap` implementation does NOT yet create a worktree per role. Adding that is a trivial loop over `role.branch` → `addWorktree` + adjusting the spawn cwd. Flagged for Phase 6 when durability wraps everything together.
2. `spawn_ssh_domain` remains a 501 stub (deferred per `v3.0-decisions.md` §8).
3. `/api/personas` response is ~100KB for 65 personas; fine for localhost but candidate for caching headers in Phase 9.
