# theorchestra v3.0 — Specification

- **Status**: Draft (2026-04-19)
- **Baselines**: `docs/v3.0-decisions.md` (locked decisions), ADR-001 through ADR-005
- **Supersedes**: v2.7.0 (stays running in parallel until cutover — §Migration)

## 1. Goal

Rebuild theorchestra as a portable, always-on AI-orchestration substrate. v3.0 replaces WezTerm-hosted panes with PTY-backed panes rendered inside a single dashboard web app. The user coordinates multiple Claude / Codex / gemini-cli / droid / opencode sessions from any device (desktop, laptop, phone) without losing continuity.

This spec captures functional and non-functional requirements, maps them to release criteria, and lists what's deferred.

## 2. Users and use cases

Single user: the operator ("user"). No multi-user, no tenancy. Primary use cases:

1. **Long-lived parallel agents.** Run 4-10 Claude / Codex panes simultaneously; each is a specialised persona (reviewer, coder, frontend-design, backend-dev, etc.). Coordinate via A2A.
2. **Device-switch continuity.** Start a pane on a desktop; an hour later, resume it on a phone via cloudflared URL.
3. **Active orchestration.** Omniclaude autonomously handles mechanical events (OK CONTINUE, auto-handoff, ack, dispatch) and asks the user only on content decisions.
4. **Roadmap-driven dispatch.** User writes `vault/_roadmap.md`; omniclaude queues tasks from it into the right panes and tracks them in `active_tasks.md`.
5. **Handoff + ctx-reset.** When a pane fills, omniclaude writes a structured handoff file and reloads the pane into a fresh context seamlessly.

## 3. Scope

### In scope (MVP)

- PTY-backed pane substrate (ADR-001)
- Vite + React + `@wterm/react` frontend (ADR-002)
- Active-orchestrator omniclaude with agent-browser a11y + SSE event input (ADR-003)
- PTY durability: browser-close + dashboard-restart + host-reboot survival (ADR-004)
- Bearer-token auth over cloudflared (ADR-005)
- All 16 MCP tools with v2.7.0 signature parity (see §FR-MCP)
- All 12 v2.x carryover features (§FR-Carryover)
- Dashboard inline chat between user and omniclaude
- Telegram push-notification-only (no live streaming)
- `npm install -g theorchestra` + `theorchestra start` single-command install

### Out of scope (v3.0 Phase 2 backlog)

From `v3.0-decisions.md` §8:

- SSO via Cloudflare Access
- Tailscale mesh as a cloudflared alternative
- Multi-machine / remote PTY backends over SSH
- Full systemd-style supervisor (dashboard-process-crash recovery)
- `Task` tool (subagent) integration

## 4. Functional requirements

### FR-PTY — PTY substrate

| ID | Requirement |
|---|---|
| FR-PTY-1 | Dashboard process spawns node-pty PTYs with configurable shell, cols, rows, cwd, env. ConPTY on Windows. |
| FR-PTY-2 | Each PTY is addressed by a durable `session_id` (UUIDv4). |
| FR-PTY-3 | Backend maintains a ring buffer of the last 10k lines per pane (configurable). |
| FR-PTY-4 | Frontend connects via WebSocket using xterm.js `AttachAddon`; attaches bidirectionally unless read-only flag set. |
| FR-PTY-5 | Resize events propagate frontend → backend → PTY. |
| FR-PTY-6 | Backpressure: backend honours XOFF/XON (`handleFlowControl: true`) to prevent PTY overrun. |

### FR-UI — Frontend

| ID | Requirement |
|---|---|
| FR-UI-1 | Vite + React + TypeScript (strict mode). |
| FR-UI-2 | Feature parity with v2.7 dashboard (`src/dashboard.html`): 4 tabs (Sessions, Live, Desktop, Spawn), right sidebar 3 reorderable panels (OmniClaude monitor, A2A activity, compact Events), bottom Active Tasks strip. |
| FR-UI-3 | New: dashboard-inline chat panel (user ↔ omniclaude). Messages and history persist server-side. |
| FR-UI-4 | Spawn wizard covers 95+ personas (`~/.claude/agents/`), git-worktree option, permission-mode selector, persona badge preview. |
| FR-UI-5 | Persona badges in pane tab titles: `[persona-name]`. |
| FR-UI-6 | SSE event stream drives live updates; no polling from the frontend. |
| FR-UI-7 | Auth: `/login` route collects bearer token, stores in localStorage, attaches to every request + WS upgrade. |
| FR-UI-8 | Mobile-responsive: phone viewport renders usable dashboard (pane view + chat). |

### FR-MCP — MCP tool surface (16 tools, v2.7.0 parity)

Tools must have identical signatures to v2.7.0 `src/mcp-server.cjs`:

| # | Tool | Purpose |
|---|---|---|
| 1 | `discover_sessions` | Enumerate live panes with project, persona, status |
| 2 | `read_output` | Read scrollback (last N lines) |
| 3 | `send_prompt` | Send text to pane (no trailing enter) |
| 4 | `send_key` | Send single key (`enter`, `1`/`2`/`3`, `ctrl+c`, etc.) |
| 5 | `spawn_session` | Spawn new pane. Claude path: `persona`, `cwd`, `permission_mode`, `resume`, `dangerously_skip_permissions`, `spawned_by_pane_id`, `prompt`. Generic path (P8.A): pass `cli` + `args` + `tab_title` for Codex/gemini/cmd.exe/bash/any TUI. |
| 6 | `kill_session` | Terminate pane |
| 7 | `get_status` | Return pane status (idle/working/permission/stuck) |
| 8 | `wait_for_idle` | Block until pane becomes idle or timeout |
| 9 | `list_projects` | List known projects on disk |
| 10 | `split_pane` | Split existing pane |
| 11 | `set_tab_title` | Set pane tab title |
| 12 | `spawn_ssh_domain` | Spawn pane via SSH (connects to remote host) |
| 13 | `list_workspaces` | List workspace groupings |
| 14 | `switch_workspace` | Change active workspace |
| 15 | `spawn_in_workspace` | Spawn pane into specific workspace |
| 16 | `auto_handoff` | Trigger auto-handoff sequence for a pane |

### FR-Orchestrator — Active omniclaude

| ID | Requirement |
|---|---|
| FR-Orch-1 | Omniclaude consumes SSE events: `pane_idle`, `permission_prompt`, `peer_orphaned`, `ctx_threshold`, `a2a_received`, `pane_stuck`, `task_dispatched`, `task_completed`. |
| FR-Orch-2 | Omniclaude reads pane state via agent-browser a11y-tree batch protocol. Latency target: ≤500ms for 10 panes. |
| FR-Orch-3 | Mechanics autonomy: OK CONTINUE, ack peer A2A, auto-handoff at ctx threshold, dispatch next queued task, kill stuck pane, approve routine tool-use permissions. |
| FR-Orch-4 | Content escalation: design decisions, merge, feature direction, cross-project spawns, anything marked `critical: true` in `active_tasks.md` → dashboard chat question + Telegram push. |
| FR-Orch-5 | Every decision logged to `vault/_orchestrator/decisions-<date>.md`. |
| FR-Orch-6 | Autonomy configurable per-project in `vault/_orchestrator-config.md` (denylist projects always escalate). |

### FR-Carryover — v2.x features preserved

Per `v3.0-decisions.md` §6, all 12 features must work in v3.0:

1. A2A envelope + rules (coordinator role declaration, push-vs-watch, mandatory `send_key('enter')`)
2. `/handoff` skill + structured 7-section handoff files
3. Auto-handoff daemon (30% suggest / 50% enforce, READY/NOT_READY readiness check, 120s poll, wait-for-idle before `/clear`)
4. 16 MCP tools (see FR-MCP)
5. 95+ persona pool at `~/.claude/agents/`
6. Git worktree isolation per pane
7. PRD-driven team bootstrap (YAML)
8. `[PEER-PANE CONTEXT]` bootstrap prefix (via `spawned_by_pane_id` param)
9. MemoryMaster integration (UserPromptSubmit recall, Stop auto-ingest every 15 msgs, PreCompact warn-once hook)
10. `active_tasks.md` task tracker with YAML frontmatter
11. Global `~/.claude/CLAUDE.md` + per-project `CLAUDE.md` rules
12. Telegram project topic routing (bidirectional chat; live streaming deprecated)

### FR-Persist — Durability

| ID | Requirement |
|---|---|
| FR-Persist-1 | Browser refresh → scrollback intact, cursor position preserved, input continues. |
| FR-Persist-2 | Dashboard-process restart → session manifests in `vault/_sessions/*.json` drive respawn; Claude panes use `--continue`, Codex uses documented resume flag. |
| FR-Persist-3 | Host reboot → same respawn flow. Banner: `[theorchestra] pane resumed after host restart at <ts>`. |
| FR-Persist-4 | Multi-device: two browsers on same `session_id` share output; input serialised into PTY write queue. |

### FR-Auth — Authentication

| ID | Requirement |
|---|---|
| FR-Auth-1 | Token generated on first dashboard startup, stored in `vault/_auth/token.json` (chmod 600). |
| FR-Auth-2 | Every HTTP endpoint + SSE + WS requires `Authorization: Bearer <token>` or returns 401. |
| FR-Auth-3 | Frontend stores token in `localStorage['theorchestra.token']` after `/login`. |
| FR-Auth-4 | `POST /api/auth/rotate` regenerates token and invalidates existing sessions. |
| FR-Auth-5 | Spawned panes inherit the token via `process.env.THEORCHESTRA_TOKEN`. |
| FR-Auth-6 | Rate limit: 10/min on `/api/auth/*`. |

### FR-Install — Install + launch

| ID | Requirement |
|---|---|
| FR-Install-1 | `npm install -g theorchestra` installs the CLI globally. |
| FR-Install-2 | `theorchestra start` launches dashboard server + PTY backend in one process. |
| FR-Install-3 | `theorchestra start --with-telegram` additionally boots the Telegram push-notifier child process. |
| FR-Install-4 | `theorchestra rotate-token` rotates the auth token. |
| FR-Install-5 | `theorchestra stop` cleanly terminates PTYs (SIGTERM, then SIGKILL after 5s) and the dashboard. |
| FR-Install-6 | node-pty native binary shipped via `prebuild-install` for Windows x64, macOS arm64/x64, Linux x64. No C++ toolchain required. |

## 5. Non-functional requirements

### NFR-Perf — Performance

| ID | Requirement |
|---|---|
| NFR-Perf-1 | Pane output → browser render latency: p95 ≤200ms local, ≤1000ms via cloudflared. |
| NFR-Perf-2 | SSE event → omniclaude reaction (decision logged): p95 ≤5s. |
| NFR-Perf-3 | agent-browser batch read of 10 panes: ≤500ms. |
| NFR-Perf-4 | Frontend bundle size: ≤500KB gzipped. |
| NFR-Perf-5 | Memory ceiling: 20 concurrent panes × 10k-line ring buffer ≈ 40MB. Total dashboard RSS ≤500MB under that load. |

### NFR-Reliability

| ID | Requirement |
|---|---|
| NFR-Rel-1 | 24h continuous run with 4+ active panes, no process crash, no pane loss. |
| NFR-Rel-2 | 10054 error class absent: zero occurrences in 24h run (should be impossible-by-construction per ADR-001). |
| NFR-Rel-3 | Reconnect storm: 10 browser-refreshes in 60s, no backend degradation. |

### NFR-Security

| ID | Requirement |
|---|---|
| NFR-Sec-1 | No unauthenticated endpoints, including SSE and WS. |
| NFR-Sec-2 | Token is ≥256 bits of entropy, generated via CSPRNG. |
| NFR-Sec-3 | Token file has OS-level permissions restricting read to the current user. |
| NFR-Sec-4 | No token leakage in logs, decision files, or SSE payloads. |
| NFR-Sec-5 | MCP tools validate all input; no shell-injection surface. |

### NFR-Portability

| ID | Requirement |
|---|---|
| NFR-Port-1 | Works on Windows 10+, macOS 13+, Linux (x64, glibc 2.28+). |
| NFR-Port-2 | Dashboard renders usably on Chrome / Safari mobile (viewport ≥375px width). |
| NFR-Port-3 | cloudflared tunnel supported; no assumption of a static public IP. |

### NFR-Observability

| ID | Requirement |
|---|---|
| NFR-Obs-1 | Every orchestrator decision in append-only `vault/_orchestrator/decisions-<date>.md`. |
| NFR-Obs-2 | SSE event schema documented; every event type has a JSON schema checked at emit time in dev mode. |
| NFR-Obs-3 | Dashboard has a `/debug` route showing live SSE feed, pane registry, PTY stats. |

## 7. Migration

Per `v3.0-decisions.md` §7 — parallel dual-run, gradual cutover:

1. v2.7.0 stays running on WezTerm. Tag already pushed (commit f59236b on origin/main).
2. v3.0 develops on a new `v3.0` branch off main.
3. v3.0 uses port **:4300** while in dogfood (v2.7.0 keeps :4200).
4. Dogfood v3.0 pane-by-pane for 1-2 weeks.
5. Once all panes migrated, stop v2.7.0. WezTerm optional thereafter.
6. v3.0 takes over :4200 at cutover.

Reversible at every step.

## 8. Open questions (to resolve during build)

These are non-blocking but will be pinned down during the relevant phase:

1. **State management library**: Zustand vs. React Context (decide in Phase 2).
2. **agent-browser distribution**: include Rust binary in the npm package (via `prebuild-install`) vs. require separate install? (decide in Phase 3)
3. **Ring buffer overflow policy**: drop oldest lines vs. chunk to disk? (MVP default: drop; decide in Phase 6 if any user scrollback need exceeds 10k lines)
4. **Multi-device input arbitration**: today's draft is "both devices' input interleaves in PTY write queue". Enough? (revisit in Phase 6 after phone testing)
5. **Codex resume flag**: document the exact resume command Codex ships; record in the respawn registry (Phase 6).
