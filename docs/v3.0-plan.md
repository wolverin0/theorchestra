# theorchestra v3.0 — Phased Implementation Plan

- **Status**: Draft (2026-04-19)
- **Baselines**: `docs/v3.0-spec.md`, ADR-001 through ADR-005, `docs/v3.0-decisions.md`
- **Branch**: `v3.0` (off `main`, diverges from commit f59236b)
- **Port during dogfood**: :4300 (v2.7.0 keeps :4200)

## Strategy

Thin vertical slices. Each phase stands up a working end-to-end path for one subsystem, ships with a verification gate, and commits on the `v3.0` branch. No phase starts without the prior phase's gate passing. Parallel work is allowed within a phase (tracked as subtasks); cross-phase parallelism is not allowed until Phase 6.

- **Definition of "gate passed"**: every acceptance check in the phase gate executes (test or documented manual procedure) and produces the expected result. Documented in a `docs/v3.0-phase-<N>-gate.md` artifact with paste-able command output.
- **Definition of "thin slice"**: the smallest scope that lets us exercise the subsystem end-to-end for real.
- **Rollback criterion**: every phase must leave the v3.0 branch in a runnable state. If a phase goes more than 2 working days over estimate without a runnable state, split the phase.

Estimate notation: "S" = ≤1 day, "M" = 2-4 days, "L" = ≤1 week. Estimates are effort, not wall-clock.

## Phase 1 — Substrate core

**Goal.** One Claude Code pane spawns inside the dashboard process via node-pty, renders in a browser tab via xterm.js, accepts input, survives a browser refresh.

**Deliverables.**
- `src/backend/pty-manager.ts` — node-pty spawn + ring buffer (10k lines) + data broadcast
- `src/backend/ws-server.ts` — WebSocket endpoint `/ws/pty/:session_id`, xterm `AttachAddon`-compatible wire protocol
- `src/frontend/` — Vite + React + TS scaffolding with one `<Terminal />` (@wterm/react) component on a `/` route
- `scripts/theorchestra-start.ts` — launches backend (port :4300)
- Minimal session registry (in-memory `Map<session_id, PtyHandle>`)

**Depends on.** Nothing. Branch creation: `git checkout -b v3.0`.

**Gate (Phase 1). STATUS: PASSED 2026-04-19 — see `docs/v3.0-phase-1-gate.md`.**
- [x] `theorchestra start` launches on :4300
- [x] Visiting `http://localhost:4300` shows one xterm rendering a live process (default is `cmd.exe`/`bash`; `claude` behind `THEORCHESTRA_DEFAULT_CLI=claude` — see gate doc §Deviation)
- [x] Typing into xterm writes to the process, output renders back
- [x] Browser refresh: scrollback restored via backend ring-buffer replay
- [ ] 24h process uptime with 1 active pane, zero crashes, zero 10054 events — DEFERRED to Phase 10 dogfood (cannot run soak in build session). Zero-10054 holds by construction per ADR-001.
- [x] Gate doc: `docs/v3.0-phase-1-gate.md`

**Estimate.** M (2-4 days).

## Phase 2 — MCP tool parity

**Goal.** All 16 MCP tools (FR-MCP) reimplemented against the node-pty backend with v2.7.0 signature compatibility. Existing Claude Code sessions can use `mcp__wezbridge__*` tools pointed at v3.0.

**Deliverables.**
- `src/mcp/server.ts` — MCP server emits the 16 tools from `src/mcp-server.cjs`, identical schemas
- Each tool implementation hits the dashboard backend over loopback HTTP (reusing the token, so MCP tools are just thin clients of the HTTP API)
- Tool-by-tool integration tests: spawn → send_prompt → read_output → kill_session per tool

**Depends on.** Phase 1.

**Gate (Phase 2). STATUS: PASSED 2026-04-19 — see `docs/v3.0-phase-2-gate.md`.**
- [x] `discover_sessions` returns live pane list (evidence: backend + MCP gate)
- [~] `spawn_session { persona: 'reviewer' }` opens a new pane with the reviewer persona preloaded — partial: wiring complete (persona resolved + `--append-system-prompt-file` passed + record tracks persona field); full Phase-5 verification deferred
- [x] `send_prompt` + `send_key('enter')` delivers a prompt — evidence in both gate harnesses; note v3.0 `send_prompt` already appends `\r`, so the extra enter is belt-and-suspenders
- [~] `auto_handoff { pane_id }` — MCP tool registered and returns structured `pending_phase_4` payload; full impl is Phase 4 dependency
- [x] Every tool has a round-trip test — 11 backend + 5 MCP assertions, 16/16 passing
- [ ] Regression sweep against v2.7 Claude pane reconfigured to v3.0 MCP — DEFERRED to Phase 10 dogfood
- [x] Gate doc: `docs/v3.0-phase-2-gate.md`

**Estimate.** L (≤1 week).

## Phase 3 — Agent-browser + SSE event bus

**Goal.** Replace `omni-watcher.cjs`'s 30s regex-scrollback poll with agent-browser a11y-tree reads + push-based SSE events. Omniclaude can subscribe to the event bus.

**Deliverables.**
- `src/backend/events.ts` — canonical SSE event schema: `pane_idle`, `permission_prompt`, `peer_orphaned`, `ctx_threshold`, `a2a_received`, `pane_stuck`, `task_dispatched`, `task_completed`
- `src/backend/event-emitters/*.ts` — one per event type, reading from the PTY data stream, ring buffer, or `active_tasks.md` watcher
- agent-browser Rust CLI integration: sidecar spawn on startup, stdin/stdout JSON batch protocol
- Client library for agent-browser: `src/backend/agent-browser-client.ts` with `readPane(session_id)` and `readAll([session_id...])`
- a11y mode enabled on every `<Terminal />` with stable DOM id `pane-<session_id>`

**Depends on.** Phase 1 (PTY data stream) and Phase 2 (session registry).

**Gate (Phase 3).**
- [ ] All 8 SSE event types fire within 1s of the triggering data (e2e test with scripted PTY input)
- [ ] agent-browser batch read of 10 simulated panes completes in ≤500ms
- [ ] a11y read integrity contract: feed a canned PTY stream, read produces a known text output (golden file)
- [ ] `omni-watcher.cjs` is removed from the v3.0 branch (or marked legacy; not spawned by `theorchestra start`)

**Estimate.** L (≤1 week). Agent-browser distribution is the biggest unknown; if blocked, treat the Rust binary as a user-installed prerequisite and defer packaging to Phase 10.

## Phase 4 — Auto-handoff + A2A protocol carryover

**Goal.** Carry over the v2.x A2A envelope rules and auto-handoff daemon on the new substrate.

**Deliverables.**
- `src/backend/a2a.ts` — envelope parser (`[A2A from pane-X to pane-Y | corr=... | type=...]`) feeding the `a2a_received` SSE event, tracking open corr IDs and emitting `peer_orphaned`
- `src/backend/auto-handoff.ts` — daemon honouring the 30% suggest / 50% enforce thresholds; READY/NOT_READY readiness-check protocol (120s poll); wait-for-idle before `/clear`; Ctrl+C + double-enter
- UI: handoff suggest toast (30%) and enforce modal (50%) — React components
- `/handoff` skill invocation surface preserved; external-write fallback for ctx-full panes
- Per-project denylist respected (`vault/_orchestrator-config.md`)

**Depends on.** Phase 2 (MCP tools for send_prompt/send_key, auto_handoff), Phase 3 (ctx_threshold event).

**Gate (Phase 4).**
- [ ] 3 simulated high-ctx panes run the auto-handoff end-to-end, producing `<source-cwd>/handoffs/handoff-to-<target>-<ts>-<uuid>.md` with the 7-section template
- [ ] A2A envelope: send a request from pane-A to pane-B, receive progress + result, track corr correctly
- [ ] Orphan detection: kill pane-B mid-exchange; `peer_orphaned` emits for pane-A within 5s
- [ ] Dual-enter rule (mandatory `send_key('enter')` after `send_prompt`) enforced at the MCP layer; test asserts double-enter on no-response

**Estimate.** M (2-4 days).

## Phase 5 — Agency mode

**Goal.** Persona-specialised pane spawn, git-worktree isolation, PRD-driven team bootstrap, `[PEER-PANE CONTEXT]` bootstrap prefix.

**Deliverables.**
- `src/backend/personas.ts` — reads `~/.claude/agents/` (Claude) + equivalent Codex source; returns persona system prompt to inject at spawn time
- `src/backend/worktree.ts` — on `spawn_session { worktree: true }`, runs `git worktree add <path> <branch>` in the target repo before spawning
- `src/backend/prd-bootstrap.ts` — reads a PRD YAML (role → task → persona), spawns the whole team in parallel
- `[PEER-PANE CONTEXT]` header auto-injection when `spawned_by_pane_id` is provided (copy v2.7 implementation)
- Spawn wizard UI: persona picker (95+ choices), permission-mode selector (plan / acceptEdits / default / bypassPermissions), worktree toggle

**Depends on.** Phase 2 (spawn_session), Phase 4 (A2A for peer-context addressing).

**Gate (Phase 5).**
- [ ] Spawn wizard UI lists all personas from `~/.claude/agents/`
- [ ] Worktree spawn: two panes on the same repo at different branches, neither sees the other's edits
- [ ] PRD bootstrap: a 3-role YAML spawns 3 panes with correct personas and pane tab titles `[role-name]`
- [ ] `[PEER-PANE CONTEXT]` prefix appears in the spawned pane's first message with correct pane_id and coordinator pane_id

**Estimate.** M (2-4 days).

## Phase 6 — PTY durability

**Goal.** Satisfy FR-Persist end-to-end: browser refresh, dashboard restart, host reboot, multi-device.

**Deliverables.**
- `src/backend/session-manifest.ts` — writes `vault/_sessions/<session_id>.json` on spawn and status change
- On dashboard startup: scan manifests, respawn panes where PTY is dead (all panes on Windows; POSIX may reattach where possible — best-effort)
- Respawn registry: `persona → resume command` mapping (Claude: `claude --continue`, Codex: documented resume flag, others: fresh)
- xterm SerializeAddon integration on the client for graceful refresh
- Multi-device: broadcast PTY data to all WS clients on a session; input write-queue serialises from either

**Depends on.** Phase 1 (PTY lifecycle), Phase 5 (persona registry).

**Gate (Phase 6).**
- [ ] Browser refresh: 50 lines of output survive refresh with cursor intact
- [ ] Dashboard restart: long Claude conversation resumes via `--continue`, cwd preserved, first prompt answered from conversation history
- [ ] Host reboot: same as dashboard restart, but after `shutdown /r`. Respawn banner visible
- [ ] Multi-device: desktop + phone connect to same session_id, share output both ways, input from either lands

**Estimate.** M-L (≥4 days). Windows PTY reattach behaviour is the unknown; if infeasible, fall back to respawn-only and document.

## Phase 7 — Active-orchestrator omniclaude

**Goal.** Omniclaude consumes the SSE event bus + agent-browser reads, runs the mechanics-autonomous / content-asks split. User-facing decision log in `vault/_orchestrator/decisions-<date>.md` carries over.

**Deliverables.**
- `vault/_orchestrator-worker/CLAUDE.md` — updated worker contract referencing the new SSE schema and a11y batch reads (replaces v2.x JSON I/O contract where applicable)
- `src/backend/orchestrator-executor.ts` — autonomous action execution (kill, continue, ack, dispatch, auto-handoff trigger)
- Classifier: every action passes through `classifyAction()` — mechanics/content split, with the v2.x safety rails (cooldown, loop detection, destructive keyword scan, denylist, no-self-actions) preserved
- Dashboard chat panel: omniclaude asks user via `/api/chat/ask`; user answers inline; answer routes back to omniclaude
- Telegram push-notification for content decisions: "omniclaude has a question, open dashboard"

**Depends on.** Phase 3 (events + a11y), Phase 4 (A2A), Phase 6 (pane lifecycle stable).

**Gate (Phase 7).**
- [ ] 10 consecutive "OK CONTINUE" cases handled autonomously across 1 day of dogfood (decision log confirms)
- [ ] 3 content decisions correctly routed to dashboard chat + Telegram push, user answered, omniclaude resumed on the user's decision
- [ ] v2.x safety rails preserved: cooldown (max 1 auto-continue per 90s per pane), loop detection (3x same action → pause), destructive keyword scan blocks `rm -rf` + `drop` + `push` etc.
- [ ] Denylist project cannot be auto-continued (always escalates)

**Estimate.** L (≤1 week).

## Phase 8 — Dashboard chat + Telegram push-only

**Goal.** v2.x Telegram live-streaming deprecated; Telegram is push-notification only. User ↔ omniclaude chat is primary in dashboard.

**Deliverables.**
- `src/frontend/components/Chat/*` — chat panel with history persisted in `vault/_chat/history.json`
- `src/backend/telegram-push.ts` — replaces `telegram-streamer.cjs` semantics: no live stream, only notification events ("question waiting", "task completed", "pane crashed")
- Bidirectional: user can reply to the notification in Telegram; reply routes to the paired dashboard chat context
- Per-project topic routing preserved (`~/.omniclaude/telegram-topics.json`)
- v2.x `telegram-streamer.cjs` removed from v3.0

**Depends on.** Phase 7.

**Gate (Phase 8).**
- [ ] Dashboard chat: user types, omniclaude answers, history persists across refresh
- [ ] Telegram push: content decision triggers a push; user replies in Telegram; reply appears in dashboard chat
- [ ] No live-streaming traffic from panes to Telegram (network trace confirms zero outbound from the stream path)

**Estimate.** M (2-4 days).

## Phase 9 — Auth + install flow

**Goal.** FR-Auth + FR-Install end-to-end. `npm install -g theorchestra` from npm (publish to a private registry or unscoped with a dummy version for dogfood), `theorchestra start`, bearer token flow, cloudflared tested end-to-end.

**Deliverables.**
- `src/backend/auth.ts` — token generation, storage (`vault/_auth/token.json` chmod 600), validation middleware, rotate endpoint
- `/login` route in the frontend, localStorage token storage, bearer header attachment
- Token injection into spawned panes via `THEORCHESTRA_TOKEN` env var
- CLI: `theorchestra start`, `theorchestra stop`, `theorchestra rotate-token`
- npm package manifest: `bin`, `prebuild-install` for node-pty native binaries (Windows x64, macOS arm64/x64, Linux x64)
- `docs/install.md` — cloudflared setup instructions

**Depends on.** Phase 2 (HTTP API surface), Phase 6 (stable pane lifecycle). Can run in parallel with Phase 8.

**Gate (Phase 9).**
- [ ] Fresh machine: `npm install -g theorchestra` succeeds, `theorchestra start` boots in ≤30s
- [ ] 401 on missing / invalid bearer for every `/api/*` endpoint, SSE, and WS upgrade
- [ ] Rotate: rotate token, old browsers get 401, must re-auth
- [ ] Cloudflared: tunnel up, paste URL on phone, paste token, full dashboard renders, can send input to a pane
- [ ] MCP inheritance: spawned pane's MCP tools succeed (token read from env)

**Estimate.** M-L (4-6 days). The packaging/prebuild-install story is the unknown.

## Phase 10 — Dogfood, parity audit, cutover

**Goal.** Verify every acceptance criterion from `v3.0-spec.md` §6; migrate user's daily workflow off v2.7.0 and onto v3.0.

**Deliverables.**
- Parity checklist: each of the 12 v2.x carryover features checked end-to-end (test or documented manual repro)
- 24h continuous dogfood: 4+ panes, log-scrape confirms no crashes + zero 10054 events
- Phone dogfood: one session driven entirely from phone for ≥1 hour
- Auto-handoff dogfood: 3 real high-ctx panes pass the full flow
- Omniclaude autonomy dogfood: 10 autonomous + 3 escalated content decisions, logged
- Release notes: `CHANGELOG.md` with v3.0 entry citing all ADRs and this plan
- Cutover: stop v2.7.0 processes, uninstall WezTerm (optional), flip v3.0 port from :4300 → :4200
- Tag: `v3.0.0`, push to origin/main (after merging `v3.0` branch)

**Depends on.** Phases 1-9.

**Gate (Phase 10 = release gate).**
- [ ] All 11 acceptance criteria in `v3.0-spec.md` §6 verified with paste-able evidence
- [ ] `docs/v3.0-phase-10-gate.md` lists each criterion → evidence
- [ ] v3.0 branch merged to main, tagged `v3.0.0`
- [ ] v2.7.0 stopped; v3.0 owns :4200

**Estimate.** M-L (≤1 week). Most of the cost is calendar time for the 24h dogfood run, not effort.

## Risk register

| Risk | Mitigation |
|---|---|
| node-pty reattach infeasible on Windows across dashboard restart | Fall back to respawn-only (Phase 6); document in ADR-004. Reboot story unchanged. |
| agent-browser Rust CLI packaging friction | Ship as a user-installed prerequisite (Phase 3); revisit npm bundling post-MVP. |
| Codex resume flag unclear / unstable | Record exact resume command per-CLI in the respawn registry; default to fresh respawn if unknown. |
| Feature parity with 5000-line monolith incomplete | Maintain a feature inventory: one row per v2.7 capability → v3.0 component → test. Track in Phase 10 checklist. |
| `@wterm/react` maturity concerns | Fallback: use xterm.js directly via thin React wrapper; retain `@wterm/react` preference when stable. |
| 24h dogfood surfaces a crash class not covered by tests | Treat as a regression, file per-phase; do not release until cleared. |

## Parallelism plan

Phases 1 → 2 → 3 strict sequence. From Phase 4 onward:

- Phase 4 (A2A + auto-handoff) and Phase 5 (agency mode) can overlap after Phase 3.
- Phase 6 (durability) must follow Phase 5 (needs persona registry).
- Phase 7 (orchestrator) must follow Phase 6.
- Phase 8 (chat + Telegram) and Phase 9 (auth + install) can overlap after Phase 7.
- Phase 10 (dogfood) is strictly last.

Per `docs/v3.0-decisions.md` coordination rules: if the user spawns a peer pane to parallel-develop, the coordinator MUST declare role (parallel-worker / qa-verifier / pre-stager / monitor-only). Default for v3.0 build is **qa-verifier on each phase gate** — the executor pane runs Phase N, the coordinator validates the gate.
