# Review — `landing.html` + `sliders.css`

## Summary

Reviewed a two-file landing-page designer playground: a 305-line `landing.html` (Hero, Features, Testimonials, Pricing, Footer + a sticky control panel with 10 live sliders, theme toggle, reset, JSON export) and a 199-line `sliders.css` of design tokens, custom range-thumb styling, button/card utilities, and a `prefers-reduced-motion` block. Overall structure is clean — semantic `<section aria-labelledby>` landmarks, implicit `<label>`→`<input>` associations, a skip link, and a well-scoped IIFE wiring sliders ↔ CSS custom properties ↔ localStorage. The standout is the JSON export (`currentSnapshot()` at landing.html:250) which captures `{version, exportedAt, theme, cssCustomProperties}` — a shippable design-tokens artefact, not a toy dump. **However, there is a critical token-naming mismatch between the two files that visually breaks the page on first paint** (detailed in Technical #1), which must be fixed before this is demo-ready.

## Design feedback

- **Visual hierarchy is destroyed by the broken body background.** Because of the token-prefix mismatch (see Technical #1), `body { background: var(--neutral-bg); color: var(--neutral-fg); }` at landing.html:14-15 resolves to invalid → the browser falls back to UA defaults (white on black). Pricing tiles and the hero-tint gradient (`linear-gradient(135deg, color-mix(... var(--primary) 12% ...))` at landing.html:47) also evaporate. The `.card` utility in sliders.css:108 still works because *it* references the correct `--color-neutral-bg`, so cards render on a white inline background — there is no brand tint anywhere on the page. The page effectively has no accent color until this is patched.
- **Responsive behaviour has a dead zone between 1024 px and ~1300 px.** The layout breakpoint at landing.html:20 pops the 340 px control panel into a sidebar starting at `min-width: 1024px`, but the feature/testimonial/pricing grids use `minmax(220px, 1fr)` / `minmax(240px, 1fr)` (landing.html:55, 61, 71). At 1024-1100 px viewport, the remaining preview column is only ~640 px, so each grid collapses to 2 or even 1 column, making the pricing row especially cramped. A `@media (min-width: 1280px)` guard — or reducing sidebar to 300 px — would give breathing room.
- **Slider UX feels great but lacks step parity.** Hue slider (landing.html:197) goes 0-360 in steps of 1 — fine. But `--hero-pad` 24-160 step 2 (line 173) vs `--features-gap` 8-64 step 1 (line 181) vs `--hero-fs` 24-96 step 1 (line 175) — inconsistent step granularity means dragging feels different per slider. Either align all to step=1, or pick step values proportional to the range (e.g. step=2 for ranges >80 units).
- **Theme toggle causes a FOUC on systems that prefer dark.** The `<html>` element hardcodes `class="theme-light"` (landing.html:2), and the `@media (prefers-color-scheme: dark)` block at sliders.css:26 is overridden by the class (sliders.css:38 `.theme-light` resets the palette to the light values). So a user with `prefers-color-scheme: dark` sees a flash of light theme until the IIFE reaches `applyTheme(savedTheme || (prefersDark ? 'dark' : 'light'))` on line 281 and swaps the class. A tiny blocking `<script>` in `<head>` (or a `color-scheme: light dark` hint on `:root`) would eliminate the flash.

## Technical feedback

- **[CRITICAL] Token-naming mismatch between inline CSS and `sliders.css`.** The inline `<style>` block in `landing.html` uses unprefixed names: `var(--neutral-bg)` (line 14), `var(--neutral-fg)` (line 15, 28, 35, 87), `var(--primary)` (line 47, 91, 96), `var(--accent)` (line 66). But `sliders.css:1-24` declares them with a `--color-` prefix: `--color-neutral-bg`, `--color-neutral-fg`, `--color-primary`, `--color-accent`. None of the unprefixed references resolve, so every `var()` with those names falls back to its initial value. Fix by either renaming the CSS tokens (drop the `--color-` prefix) or updating the inline style to use `--color-*`. This is a single search-and-replace but it silently breaks ~8 property declarations. Note `--fs-body`, `--section-pad`, and `--card-gap` *do* match and work correctly, which is why the bug is easy to miss at a glance.
- **CSS token architecture mixes two sizing families for the same concept.** `sliders.css:10` defines `--fs-hero: 3rem` (intended as the typographic scale), but the hero's actual font-size at landing.html:49 is driven by `--hero-fs` (set dynamically in px by the slider, default 56). The `--fs-hero` token is never referenced in markup — it is dead weight. Either drop it, or rewire the slider to output `rem` values and overwrite `--fs-hero` directly. Same story for `--fs-h2`/`--fs-h3` (sliders.css:11-12) — declared, never consumed.
- **Dead selector `.slider-row` (sliders.css:169-183) never matches.** The control panel renders each slider inside a `<label>` with nested `<span class="row">` (landing.html:172-174). There is no element with class `slider-row`, so the 15-line grid layout block is unreachable. Either migrate the label markup to `<div class="slider-row">` or delete the rule.
- **JS structure is solid, two small nits.** `persist()` at landing.html:228 writes localStorage on *every* `input` event — dragging a slider dispatches 50-200 events, each doing `JSON.stringify` + synchronous storage write. Debounce with `requestAnimationFrame` or a 100 ms trailing timer. Second: `STORAGE_KEY = 'landing-designer:v1'` and `THEME_KEY = 'landing-designer:theme'` (lines 214-215) are stored as two separate entries — collapsing them into one snapshot would halve the I/O and make version migrations easier.
- **Persistence + export logic is mostly sound but drops units on the JSON round-trip.** `currentSnapshot()` (line 250) reads `s.dataset.unit` to append the unit at export time — good. But on `restore()` (line 234), the saved snap value is a unitless string and is assigned back to `s.value` (which expects a raw number) — also good. However, if someone hand-edits an exported JSON and re-imports it (there is no import button today, but that is the natural next feature), the `cssCustomProperties` object has values *with* units (`"72px"`) and the sliders expect *without*. Worth a `parseFloat()` shim on the eventual import path.
- **The try/catch around localStorage (lines 231, 236, 279, 285) swallows errors into an empty `(_) {}`.** Safari Private Browsing throws `QuotaExceededError` on `setItem`. A silent failure means a user tweaks sliders for 10 minutes, reloads, and loses everything with no signal. At minimum log to `console.warn`; ideally show a one-time non-blocking notice.

## Accessibility audit

- **Keyboard navigation works but the focus ring on range thumbs is Chromium-flaky.** sliders.css:160-167 places `outline: 3px solid var(--color-accent)` on `::-webkit-slider-thumb:focus-visible` and `::-moz-range-thumb:focus-visible`. Firefox respects the Moz pseudo; Chromium historically ignores `outline` on the WebKit pseudo and only reliably styles via `box-shadow`. Swap to `box-shadow: 0 0 0 3px var(--color-accent)` for the WebKit branch to guarantee a visible focus ring in Chrome/Edge. Also `input[type="range"]:focus-visible { outline: none }` (line 159) strips the element-level ring before the pseudo-element one is applied — if the Chromium pseudo-outline silently fails, the slider becomes focus-invisible entirely.
- **ARIA on sliders is missing `aria-valuetext` so the unit is not spoken.** Screen readers on `<input type="range" min="24" max="160" value="72">` announce "72, slider, min 24, max 160" — no "pixels". Adding `aria-valuetext="72 pixels"` (updated in `applySlider`, landing.html:220) gives meaningful narration. The hue slider (landing.html:197) has no unit but could announce `"220 degrees, blue"` via the same hook and is a great UX bonus. Theme toggle (landing.html:165) sets `aria-pressed` (good) but the visible label is just "🌓 Theme" — a screen reader user hears "Theme toggle button, pressed" with no hint of *which* theme is active. Swap the visible text (or add `aria-label`) to `"Switch to dark theme"` / `"Switch to light theme"` that updates in `applyTheme()`.
- **Color-contrast is AA in isolation but the broken token mismatch puts the page at a coincidental pass.** With the mismatch, text falls back to CanvasText (black) on Canvas (white) = 21:1 — trivially passes. Once the mismatch is fixed, the real palette kicks in: `--color-neutral-fg #1a1d29` on `--color-neutral-bg #fafbff` = ~16:1 (AAA), `--color-muted #6b7280` on the same = ~4.6:1 (AA body). Dark-mode `--color-neutral-fg #e7e9f4` on `--color-neutral-bg #0f1220` = ~15:1 (AAA). Also note `prefers-reduced-motion` at sliders.css:191-199 zeroes transitions but the inline `.btn-primary:active { transform: translateY(1px); }` is in the landing.html `<style>` and is *not* captured by the sliders.css reduced-motion override because the override selector `*, *::before, *::after { transition: none; animation: none; }` does not touch `transform`. Add a `:active { transform: none !important; }` inside the reduced-motion block for full compliance.

## Recommended next actions

- **Fix the token mismatch first.** Either (a) in `sliders.css:1-24`, rename `--color-primary` → `--primary`, `--color-neutral-bg` → `--neutral-bg`, etc. (and propagate through sliders.css body/h1/btn/card rules), or (b) in landing.html `<style>` block, rewrite every `var(--primary)` → `var(--color-primary)` etc. Option (b) is ~8 substitutions and does not touch the shared stylesheet — preferred. Verify by opening the page and confirming the hero has its blue tint and the page background is `#fafbff` (light) / `#0f1220` (dark).
- **Add debounced persistence and a non-silent localStorage error path.** Wrap `persist()` in a 100 ms trailing timer and replace `catch (_) {}` with `catch (e) { console.warn('designer: localStorage unavailable', e); }`. Optionally surface a one-shot toast if the first `setItem` throws.
- **Tighten accessibility on sliders and theme toggle.** In `applySlider()` (landing.html:220), set `s.setAttribute('aria-valuetext', val)` alongside the readout. In `applyTheme()` (landing.html:243), update the toggle's `aria-label` to announce the action it will perform next (`"Switch to dark theme"` when currently light). And swap the WebKit focus ring from `outline` to `box-shadow` in sliders.css:160 for Chromium parity.
- **Delete dead CSS and consolidate typography tokens.** Remove `.slider-row` (sliders.css:169-183), `--fs-hero` / `--fs-h2` / `--fs-h3` (sliders.css:10-12) unless you wire sliders to them, and the duplicated `@media (prefers-color-scheme: dark)` block (sliders.css:26-36) that is made redundant by the `.theme-dark` class override on lines 48-56 — keep the class-based overrides as the single source of truth and reduce the `prefers-color-scheme` block to just setting the default class if you want system-preference bootstrapping without JS.
- **Add a responsive guard at the 1024-1280 px band.** Either bump the sidebar breakpoint to `min-width: 1280px` at landing.html:20, drop the panel width from 340 px to 300 px, or lower the grid `minmax` from 240 px → 200 px for testimonials and pricing. Test at 1024, 1152, and 1280 to confirm pricing stays 3-up past 1280 and degrades gracefully below.
