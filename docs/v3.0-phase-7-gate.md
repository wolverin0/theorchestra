## What shipped

The active orchestrator ‚Äî rule-based decision engine that consumes the Phase 3 SSE event bus and automatically handles mechanics while escalating content decisions to a dashboard chat panel. MVP intentionally does NOT call an LLM per decision; the rules + classifier cover the plan's 8 mechanics/content targets deterministically.

### Safety rails

All active, verified by the gate:

1. **Cooldown** ‚Äî max 1 auto-continue per session per 90s.
2. **Loop detection** ‚Äî ‚â•2 recorded attempts on same `(action-kind, session)` within 5min ‚Üí 3rd blocked. (v2.x framed this as "3x ‚Üí pause"; matching intent ‚Äî the 3rd occurrence is blocked.)
3. **Destructive keyword scan** ‚Äî `continue` text or `send_key` key containing `rm -rf`, `drop table`, `git push --force`, `git reset --hard`, `migrate`, `deploy`, `prod`, etc. ‚Üí forced content classification.
4. **Per-project denylist** ‚Äî reads `vault/_orchestrator-config.md`'s "Never-Auto Projects" section; listed project names force content classification.
5. **No self-actions** ‚Äî actions targeting `opts.selfSessionId` are blocked outright.

## Deviations + follow-ups

1. **No LLM calls per decision.** MVP runs deterministic rules. If a decision genuinely needs LLM judgement (rare, like novel unstructured pane output), the rule can escalate-to-user and the user's answer feeds back in via `POST /api/chat/answer`. Phase 10 dogfood will tell us if this is sufficient.
2. **Chat has no SSE push yet.** The dashboard will poll `GET /api/chat/messages`. Phase 8 adds `chat_updated` SSE events so the UI can react instantly.
3. **Self-session detection** ‚Äî we expose `opts.selfSessionId` but the launcher doesn't set it. If the user runs `theorchestra start` with the orchestrator *as* a pane (rather than in-process), they should pass the sessionId. Not done by default because the orchestrator in v3.0 is in-process, not in-pane.
4. **Codex resume + A2A dispatching** ‚Äî the orchestrator forwards A2A mechanics to existing flows; no new A2A writes here.
