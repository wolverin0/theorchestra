# ADR-002: Port dashboard from single-file HTML to @wterm/react + Vite

- **Status**: Accepted (2026-04-19)
- **Supersedes**: `src/dashboard.html` (≈5000-line vanilla-JS single-file UI)
- **Context anchor**: `docs/v3.0-decisions.md` §3

## Context

`src/dashboard.html` is the canonical v2.x frontend: one HTML file with inline `<script>` containing ~5000 lines of vanilla JS driving the dashboard UI. It grew organically through v2.1 → v2.7 and now carries:

- 4 top-level tabs (Sessions, Live, Desktop, Spawn)
- A right sidebar with 3 collapsible reorderable panels (OmniClaude monitor, A2A activity, compact Events)
- A bottom Active Tasks strip
- Inline SSE client, event translation layer, handoff modal, permission UI, persona picker, dashboard chat, project navigator

Pain points that block v3.0 work:

1. **No component isolation.** Every feature shares the same global DOM; regressions ripple (the v3.1 FuturaOS restyle had to patch ~30 CSS selectors to avoid breaking v2.4 panels).
2. **No HMR.** Every change requires a full browser reload; state in the dashboard (active SSE subs, filter positions, in-flight handoffs) is lost on every save.
3. **No state abstraction.** SSE events, pane state, handoff state, chat state all live as loose `let` variables on `window`. Adding orchestrator-inline-chat (§2) or a persistent Spawn-tab wizard means more globals.
4. **No build-time type checks.** Bugs in the event translation layer (`omni-watcher` → dashboard shape mapping — see `src/dashboard-server.cjs`) have shipped multiple times because there's nothing checking the payload shape.

v3.0 adds substantial new surface: an in-dashboard chat with omniclaude, a Spawn wizard that handles 95+ personas, persistent pane tabs that reconnect to PTY `session_id`s on reload, auth token flow, `@wterm/react` terminal rendering. That surface is not tractable in the current monolith.

## Decision

Port the frontend to **Vite + React + TypeScript**, using **`@wterm/react`** for terminal components. Each v2.x feature maps to a React component or route. Build output is static assets served by the same Node dashboard process (no separate frontend origin, no SSR).

Stack:
- **Vite** for dev server (HMR) and production build (rollup)
- **React 18+** for component model
- **TypeScript strict mode** for the event translation layer and MCP tool argument types
- **`@wterm/react`** for `<Terminal />` components, which wrap xterm.js + addons (fit, serialize, attach)
- **Zustand or React Context** for global state (pane registry, auth token, SSE feed) — decide at implementation time, default to Zustand
- **TanStack Query** for HTTP queries against `/api/*` endpoints
- **No Next.js.** This is an admin UI served from the same process as the backend; SSR adds zero value and doubles the install footprint.

Legacy `tmp/dashboard-react-v2.1/` is archived; v3.0 does **not** incrementally port that codebase — it starts fresh with the current v2.7 feature list as the spec (see `docs/v3.0-spec.md` §FR-UI).

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| **Keep single-file HTML, add JS modules** | Doesn't address state-management, HMR, or type-safety. Kicks the problem forward. |
| **Next.js + React** | SSR is dead weight for an admin UI served to one user. Doubles the install size and adds build complexity. |
| **SvelteKit** | Excellent DX, but `@wterm/react` does not have a Svelte equivalent; would need hand-rolled xterm bindings. |
| **Preact + HTM (no build step)** | Solves build complexity but not type-safety or component ecosystem. Nice-to-have, not sufficient. |
| **Solid.js** | Mature, but `@wterm/react` is the stated binding; retargeting to Solid means re-maintaining terminal integration. |

## Consequences

**Positive:**
- HMR during development cuts iteration cost drastically (expected: seconds per change vs. 10-20s for a full reload).
- Component boundaries let multiple contributors (or parallel agent panes) work on separate features without merge conflicts.
- TypeScript catches the `omni-watcher event shape ↔ dashboard shape` drift at compile time (`src/dashboard-server.cjs` currently has a `typeMap` that is easy to get wrong).
- `@wterm/react` gives us `<Terminal />` as a first-class component that manages xterm.js lifecycle, attach-addon wiring, and resize. No more hand-rolled `terminal.open(div)`.

**Negative / costs:**
- A full rewrite of the frontend, not an incremental migration. v2.x feature parity becomes a spec requirement (see spec §FR-UI) — we must inventory every v2.x feature before we call v3.0 shippable.
- New build toolchain. `npm run dev` → Vite. `npm run build` → rollup bundle served by the Node server.
- Initial bundle size: ≈300-500KB gzipped with React + xterm.js + addons. Acceptable for a desktop/mobile admin UI.

**Neutral:**
- Both single-file and React versions use SSE as the event bus, so `src/dashboard-server.cjs`'s event translation layer carries over with minimal changes.

## Verification

- Phase 2 exit gate: Vite dev server runs, `<Terminal />` component attaches to a node-pty `session_id` via WebSocket, echoes input back.
- Phase 8 exit gate: every v2.7 feature on the v2.x feature inventory (spec §FR-UI) has a reachable React equivalent. Screenshot diff against v2.7 for the 4 primary tabs.
- No regression in SSE event handling: the translation layer (now typed) round-trips a recorded v2.7 SSE stream with zero payload mismatches.
