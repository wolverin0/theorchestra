# ADR-005: Bearer-token auth + cloudflared-only remote access

- **Status**: Accepted (2026-04-19)
- **Context anchor**: `docs/v3.0-decisions.md` §4

## Context

v2.7.0 exposes the dashboard at `http://localhost:4200` with **no authentication**. Remote access uses **cloudflared** to tunnel that port to a public `*.trycloudflare.com` URL. The URL is random, which provides weak security-through-obscurity, but anyone who gets the URL — from a share, a screenshot, a DNS log, a browser autocomplete — has full control of every pane, which means full control of the user's shell + every AI session + every editable file on the user's machine.

The existing dashboard has no login, no token, no origin check, no CSRF protection. Every HTTP endpoint and the SSE stream are open. This was acceptable while the dashboard was localhost-only, but v3.0 assumes remote access from phones is routine (§3 portability story).

Multi-user auth is not a goal. This is a single-user tool. We need *enough* security to defeat URL discovery, not a full IDP integration.

## Decision

Adopt **bearer-token authentication** on every dashboard endpoint, paired with the existing **cloudflared tunnel** as the only supported remote-access mechanism.

### Token lifecycle

- **Generation.** On first dashboard startup (no `vault/_auth/token.json` present), the backend generates a 256-bit cryptographically random token (`crypto.randomBytes(32).toString('base64url')`) and writes `vault/_auth/token.json` (chmod 600 on POSIX; Windows ACL narrowed to the current user).
- **Rotation.** `POST /api/auth/rotate` regenerates the token; all existing sessions must re-auth. Also available via a CLI: `theorchestra rotate-token`.
- **Revocation.** Deleting `vault/_auth/token.json` and restarting rotates + invalidates all sessions.

### Token presentation

- **First visit.** The browser is redirected to `/login?return=<path>`. The user pastes the token once; we store it in `localStorage['theorchestra.token']`.
- **Subsequent requests.** The frontend attaches `Authorization: Bearer <token>` to every HTTP request and the WebSocket upgrade. Unauthenticated requests return `401 { reason: "missing_or_invalid_token" }`.
- **SSE.** The event stream requires the same header; without it, the connection is closed immediately.
- **MCP.** MCP tools running on the same machine call the backend via loopback with the token read from `vault/_auth/token.json` (same file the backend reads on startup). Tools running on peer panes inherit the token from their `process.env.THEORCHESTRA_TOKEN`, which the backend injects when spawning.

### Transport

- **Localhost:** HTTP is fine (token protects, traffic stays in-process).
- **Remote (cloudflared):** the cloudflared tunnel terminates TLS at Cloudflare's edge. Traffic between the user's device and Cloudflare is HTTPS; between Cloudflare and the user's machine, cloudflared uses its own TLS. No additional TLS layer needed for MVP.

### Origin/CSRF

- The token *is* the CSRF defence: without the bearer header, a cross-origin POST can't succeed. We do not rely on cookies, which avoids the CSRF problem entirely.
- CORS is restrictive by default: `Access-Control-Allow-Origin` matches the request's `Origin` header if and only if the bearer token validates. Otherwise, no CORS headers are emitted.

### Rate limiting

- Per-token rate limit on `/api/auth/*` endpoints (10/min). Everything else is unmetered because a valid token = trusted client.

### What we explicitly do not do

- **No OAuth.** No third-party IDP. No Cloudflare Access SSO. No user accounts.
- **No multi-user.** One token, one user, one machine.
- **No IP allowlist.** User's phone IP changes constantly; would cause more lockouts than it prevents attacks.
- **No Basic auth.** Browsers cache it badly and it shows in server logs.

## Alternatives considered

| Alternative | Why rejected (for MVP) |
|---|---|
| **Cloudflare Access SSO** | Requires a Cloudflare account with Access plan; user has to configure an IDP (Google, GitHub, etc.). Overkill for single-user. *Phase 2 candidate — §8.* |
| **Tailscale mesh, no public tunnel** | Changes the remote-access story (mesh vs. tunnel). User would need Tailscale installed on every device. Not a blocker for MVP, but shifts the install footprint. *Phase 2 candidate — §8.* |
| **Self-hosted TLS + password form** | We'd need to manage certs. cloudflared already handles TLS. Adds complexity with no security win over a 256-bit token. |
| **IP allowlist via cloudflared config** | Too brittle for mobile networks. |
| **Magic-link email** | Requires SMTP config. Token-paste is cheaper and offline-friendly. |
| **No auth, rely on URL secrecy** | Current v2.7.0 model; fails the §4 requirement to protect against URL discovery. |

## Consequences

**Positive:**
- Random URL discovery no longer grants access. An attacker who finds the cloudflared URL sees only `401 Unauthorized`.
- No new infra dependencies (Cloudflare Access, Tailscale, SMTP, etc.). Install stays as `npm install -g theorchestra` (§3).
- Token persists in browser localStorage, so re-auth on every visit is not required.
- Migration from v2.7.0 is clean: new endpoint behaviour, old clients get `401` and are told to upgrade.

**Negative / costs:**
- User must paste the token on first visit from each device. One-time friction per device.
- Token file on disk is a secret at rest. We mitigate with file-system permissions; if the user's local machine is compromised, the attacker has shell access anyway (the defence perimeter ends at `localhost`).
- No revocation per device, only global. Acceptable for single-user; if the user loses a device, rotate the token.

**Neutral:**
- MCP tool calls from peer panes keep working because the token is injected into `THEORCHESTRA_TOKEN` at spawn time. No change to the MCP API surface.
- All SSE events are now authenticated, so no "open feed" endpoints remain.

## Verification

- Phase 9 exit gate — **401 on missing token**: every `/api/*` and `/events` endpoint returns `401` without a valid bearer header.
- Phase 9 exit gate — **localStorage flow**: fresh browser → paste token → dashboard renders. Refresh → still authenticated.
- Phase 9 exit gate — **cloudflared remote**: from a phone, via cloudflared URL, paste token, full dashboard renders, SSE stream arrives, can send input to a pane.
- Phase 9 exit gate — **rotate invalidates**: rotate the token, old browsers get 401 on next request, must re-auth with the new token.
- Phase 9 exit gate — **MCP inheritance**: spawn a pane via MCP, confirm that pane's MCP tools succeed (token was injected into env).
