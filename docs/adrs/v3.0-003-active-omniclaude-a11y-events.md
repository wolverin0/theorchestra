# ADR-003: Active-orchestrator omniclaude driven by a11y-tree reads and SSE events

- **Status**: Accepted (2026-04-19)
- **Supersedes**: `src/omni-watcher.cjs`'s 30s regex-scrollback polling
- **Context anchor**: `docs/v3.0-decisions.md` §2

## Context

In v2.x, `src/omni-watcher.cjs` polls every pane every 30s by shelling out to `wezterm cli get-text`, then runs anchored regexes over the last 30 lines to classify status (idle / working / permission / stuck) and scan for A2A envelopes. This was forced up from 5s because faster polling crashed the WezTerm GUI (ADR-001). The 30s cap is the latency floor for every orchestrator reaction, including permission-prompt handling and `peer_orphaned` detection.

Regex-over-scrollback is also fragile. A status-bar line that looks like `"Ctx: 38k/200k · …"` can be partially overwritten by an in-flight render, producing bad matches. Spinner detection depends on exact glyph sequences which change between Claude Code versions. Adding a new event type means hand-tuning a new regex, which we've had to do at least four times.

Finally, v2.x omniclaude is a **passive router**: it consumes events, forwards A2A, surfaces escalations. `docs/v3.0-decisions.md` §2 escalates this role: omniclaude must decide and act on most mechanical cases without asking the user. "Send OK CONTINUE", "ack peer A2A", "kill a stuck pane", "dispatch next queued task" — all autonomous. Content decisions (design choices, cross-project spawns, merges, feature direction) still ask.

Active autonomy needs two inputs that v2.x cannot reliably provide:

1. **Deterministic pane-state reads** (not regex): what is the pane doing *right now*?
2. **Push-based semantic events** (not poll): the orchestrator reacts in seconds, not the 30s poll window.

## Decision

v3.0 replaces the scrollback poll with a two-track input pipeline:

### Track 1 — agent-browser a11y-tree reads

Every pane in the dashboard is an xterm.js-rendered terminal inside the browser. xterm.js ships a **screen-reader accessibility mode** that maintains an ARIA tree (`aria-posinset`, `aria-setsize`) synced to the terminal state, updated inside an animation frame (≤60Hz). The **agent-browser** Rust CLI reads that accessibility tree deterministically against the live DOM.

Concretely:

- Each `<Terminal />` is wrapped in a container with a stable DOM id (`pane-<session_id>`) and the a11y mode enabled at Terminal construction.
- agent-browser runs as a sidecar process (spawned by the dashboard server on startup, or on-demand) with a batch protocol: `read_pane(session_id) → { text_lines, cursor, cursor_style, last_render_ts }`.
- A batch read of all panes is a single round-trip, cheap, and does not touch the PTY stream. No scrollback-socket saturation, no 10054.
- Structured text-lines replace regex matching: status detection becomes "is the last line a known-shape status bar?", which is stable across Claude Code versions because the shape is the stable invariant, not the exact glyphs.

### Track 2 — push-based SSE events

The dashboard backend already emits SSE to the frontend. v3.0 canonicalises the event schema as the orchestrator's reactive bus. Events omniclaude subscribes to:

- `pane_idle` — PTY was writing data, now has been quiet for N seconds (backend detects on the raw data stream, not scrollback)
- `permission_prompt` — a known permission-prompt pattern was emitted; payload includes the prompt text
- `peer_orphaned` — a pane in an open A2A exchange disconnected
- `ctx_threshold` — Ctx-percentage parsed from status bar crossed 30% / 50%
- `a2a_received` — envelope parsed on output stream, payload includes `corr`, `from`, `to`, `type`
- `pane_stuck` — idle + working state held longer than configured threshold
- `task_dispatched`, `task_completed` — from `active_tasks.md` file watcher

Events are emitted with ≤1s latency from the triggering data. Omniclaude consumes the same SSE stream the dashboard UI does.

### Correlation

Omniclaude decides by correlating Track 1 and Track 2. Example: `permission_prompt` SSE arrives → omniclaude issues a `read_pane` to see the exact prompt → decides to approve/deny → sends `send_key('1')` via MCP. Total latency budget: ≤5s from prompt render to decision sent.

### Autonomy split (restated from decisions doc)

| Class | Examples | Action |
|---|---|---|
| **Mechanics** | OK CONTINUE prompts, auto-handoff at ctx threshold, A2A ack, dispatch next queued task, kill stuck pane, approve routine tool-use permissions | Autonomous; logged but not user-facing |
| **Content** | Accept/reject design decision, merge to main, feature direction, cross-project spawn, anything marked `critical: true` in `active_tasks.md` | Ask user via dashboard chat panel; Telegram push notification ("question waiting") |

Omniclaude always logs its decision with reasoning to `vault/_orchestrator/decisions-<date>.md` (v2.x audit log carries over).

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| **Keep regex-scrollback, push cadence to 5s with node-pty** | Faster poll fixes latency but doesn't fix regex brittleness or version drift. The a11y tree is structured data; scrollback is not. |
| **Parse ANSI escape sequences on the PTY stream directly** | Recreates a terminal emulator. xterm.js already does this; a11y mode is the free layer on top. |
| **OCR / vision model on rendered canvas** | Non-deterministic, expensive, worse-than-a-tree. |
| **Only SSE, no pane reads** | SSE gives event *arrival*; it can't answer "what are the exact last 10 lines right now?" for an LLM decision. Both tracks are needed. |
| **Pure-LLM autonomy (omniclaude decides everything)** | Breaks user trust. The mechanics/content split is a safety feature: merges and content direction always involve a human. |

## Consequences

**Positive:**
- Reaction latency drops from 30s floor to ≤5s for most mechanical events.
- Status detection is no longer glyph-dependent; it survives Claude Code UI refreshes.
- Omniclaude's decision log becomes more auditable because every decision cites both the SSE event that triggered it and the a11y-tree read that confirmed state.
- Agent-browser is reusable for future non-orchestrator use cases (e.g., automated regression screenshots of pane output).

**Negative / costs:**
- Two new components: agent-browser sidecar + canonical SSE event schema. Both need tests.
- xterm.js a11y mode has to stay enabled even when no user is focused on the pane; performance cost is ≤60Hz DOM writes per active pane, acceptable for ≤20 concurrent panes.
- A regression in the a11y tree (xterm.js upgrade, for instance) silently degrades omniclaude's input quality. We add a "a11y read integrity" contract test: for a canned PTY stream, the read must produce a known text output.

**Neutral:**
- v2.x Telegram bidirectional chat (§6 §12) still works; it just routes through dashboard-chat rather than streaming live pane output.

## Verification

- Phase 3 exit gate: agent-browser reads 10 panes in under 500ms per batch, with text output matching a golden file for canned PTY streams.
- Phase 3 exit gate: all eight SSE event types fire within 1s of the triggering data in an end-to-end test.
- Phase 7 exit gate: omniclaude autonomously handles 10 consecutive "OK CONTINUE" cases without escalation, and correctly escalates 3 content decisions to Telegram (per `v3.0-decisions.md` §9 release criteria).
