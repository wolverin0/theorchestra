# ADR-004: PTY durability — browser-close survives, reboot auto-respawns with `--continue`

- **Status**: Accepted (2026-04-19)
- **Context anchor**: `docs/v3.0-decisions.md` §3 (Persistence), §7 (Migration)

## Context

The portability story behind v3.0 is: user works at home on a desktop browser, closes the laptop, picks it up in a café, opens the phone browser, keeps typing into the same pane. That story dies if any of the following happen:

- Closing the browser tab kills the pane.
- Refreshing the dashboard wipes the scrollback and the pane loses its cwd / partial input.
- Rebooting the host loses every Claude / Codex session (including its conversation history).
- Reconnecting from a different device forces re-authentication to each pane individually.

v2.x accepts most of these failures because WezTerm owned pane lifetime — if WezTerm stays up, the panes stay up, but remote access (cloudflared) only exposes the dashboard, so pane access from the phone was never continuous.

v3.0 moves PTY ownership into the dashboard backend process (ADR-001). That unlocks true durability, but only if we commit to three properties.

## Decision

### Property 1 — Close browser / refresh / reconnect: pane survives

- The `<Terminal />` component addresses its backing PTY by a durable `session_id` (UUID v4 allocated at spawn time).
- PTY output is buffered in the backend in a ring buffer (last 10,000 lines by default; configurable). On WebSocket reconnect, the backend writes the serialised ring buffer to the client before streaming new data.
- The xterm.js **SerializeAddon** captures the terminal state client-side as VT sequences. On reconnect, the client replays the backend buffer into a fresh `Terminal` instance and the user lands at the current cursor position with full scrollback intact.
- `session_id → PTY` registry is in-memory on the backend. Dashboard-process restart is handled by Property 2.

### Property 2 — Dashboard process restart: PTY state persists to disk, reconnectable

- The backend writes a session manifest to `vault/_sessions/<session_id>.json` at spawn and on every status change: `{ session_id, pid, persona, cwd, cli, cli_args, spawned_at, tab_title, last_output_ts }`.
- On dashboard startup, we scan `vault/_sessions/*.json`:
  - If the `pid` is still alive AND the PTY is reattachable, reattach. (Limited: node-pty doesn't expose FD inheritance easily on Windows; on Linux we can reattach, on Windows we accept respawn per Property 3.)
  - If the `pid` is dead or reattach is not possible on this platform, fall through to Property 3.
- The ring buffer is **not** persisted across dashboard restarts in MVP. Persisting it would bloat disk; Phase 2 can add selective persistence for long-running panes.

### Property 3 — Host reboot: auto-respawn with `--continue` (Claude) / `--resume` (Codex)

- The session manifest from Property 2 survives reboot.
- On dashboard startup after a reboot, panes whose CLI supports session-resume are auto-respawned:
  - Claude Code: `claude --continue` (resumes latest in the cwd)
  - Codex: the documented resume flag (whatever Codex ships — the manifest records it)
  - gemini-cli / droid / opencode / plain bash: respawn fresh; the CLI's own conversation persistence (if any) handles continuity
- The respawn preserves `cwd`, `persona`, `tab_title`, but starts a fresh PTY PID.
- The user gets a one-line banner in each respawned pane: `[theorchestra] pane resumed after host restart at <ts>`.

### Multi-device access

- The bearer token (ADR-005) is stored in browser localStorage. On a new device, the user pastes the token once; after that, the device can reconnect to any `session_id` just like the origin device.
- Two devices on the same `session_id` share output (both receive the same WebSocket broadcast). Input from either device is serialised into the PTY write queue. (This is the same model as a shared tmux session.)

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| **systemd-style supervisor per pane** | Overkill for MVP. Adds a third process per pane. Phase 2 candidate. |
| **Persist PTY state to Redis / SQLite** | Ring buffer in memory is enough for MVP. File-system manifests are enough for cross-restart state. Adds infra dependency otherwise. |
| **Leave persistence out entirely, user respawns panes manually** | Breaks the portability story. Violates `v3.0-decisions.md` §3. |
| **Use `dtach` / `tmux` as a detachable PTY backend** | Adds an external runtime we can't control on Windows without WSL. Phase 2 alternative if node-pty reattach proves flaky. |
| **Serialize full xterm state server-side** | SerializeAddon is a client-side concern. Backend keeps the authoritative VT stream in its ring buffer; clients replay it. Single source of truth. |

## Consequences

**Positive:**
- True device-portability. The story in §3 works.
- Scrollback survives refresh (serialised VT sequences restore cursor and styling).
- Reboot no longer loses Claude conversation history (the `--continue` flag preserves it; we preserve the pane shape around it).

**Negative / costs:**
- Windows-specific: node-pty cannot reliably reattach to a running PTY across process boundaries, so dashboard-process restart on Windows = respawn (pid gone). Acceptable: the user's intent is "reboot survives"; dashboard-crash-during-a-session is rare and v2.7.0 running in parallel covers critical work (§7).
- Ring buffer memory: 10,000 lines × ~200 bytes × 20 panes ≈ 40 MB. Trivial, but we cap at 10k by default and make configurable.
- Auto-respawn assumes the CLI supports `--continue`. We keep a registry: `persona → resume command`. Unknown CLIs respawn fresh.

**Neutral:**
- v2.x `--continue` pattern for `scripts/omniclaude-forever.sh` carries over unchanged; v3.0 just generalises it to every pane.

## Verification

- Phase 6 exit gate — **browser refresh**: open a Claude pane, write 50 lines, refresh browser. The exact same 50 lines are in scrollback; cursor is at the prompt; sending new input works immediately.
- Phase 6 exit gate — **dashboard restart**: with one Claude pane in a long conversation, kill the dashboard process, restart. The pane respawns via `claude --continue`, pane cwd is preserved, first prompt after respawn is answered from conversation history.
- Phase 6 exit gate — **host reboot**: same as restart, but after a real `shutdown /r`. Respawn banner is visible. `claude --continue` picks up the last session.
- Phase 6 exit gate — **multi-device**: open a pane on desktop, open same URL + token on phone, see the same scrollback. Type from phone, output appears on desktop (and vice versa).
