# ADR-003 Addendum — Pivot from Rust `agent-browser` CLI to `@xterm/headless`

- **Date**: 2026-04-19 (Phase 3 pre-work)
- **Status**: Accepted — supersedes ADR-003 §"agent-browser a11y-tree reads"
- **Parent**: `docs/adrs/v3.0-003-active-omniclaude-a11y-events.md`

## Why the pivot

ADR-003 originally specified a Rust CLI (`agent-browser`) that would spawn a headless Chromium, connect via CDP to the dashboard's `:4300`, and read xterm.js's accessibility tree from the live DOM. That design had three problems for MVP:

1. **Distribution friction.** The Rust binary has to be prebuilt per platform (Windows x64, macOS arm64/x64, Linux x64) and bundled via `prebuild-install`. Every spawn of the sidecar is an out-of-process hop. The risk was flagged in `docs/v3.0-plan.md` Phase 3's estimate note.
2. **Runtime cost.** A headless Chromium is 150-200MB RSS, plus the Rust process, plus the socket/CDP handshake on every read. Reading 10 panes in ≤500ms (the Phase 3 gate target) is doable but wasteful.
3. **Conceptual circularity.** The dashboard *is* a browser — we don't need an external browser to read a browser. The xterm.js running in the dashboard is the same VT parser we could run server-side.

## The pivot

Use **`@xterm/headless@^6.0.0`** — the canonical headless version of xterm.js, published by the xterm.js maintainers. Same VT parser (Parser.ts, Buffer.ts, etc.), same rendering logic, no DOM.

**How it fits in the PTY pipeline:**

```
pty.onData(chunk)
  ├─→ PtyManager ring buffer (existing)
  ├─→ WebSocket broadcast to browser clients (existing)
  └─→ @xterm/headless Terminal.write(chunk)  (NEW)
```

Each managed session gets a paired `@xterm/headless` `Terminal` instance. Data chunks are written into both the ring buffer and the headless terminal. To read "what xterm would have rendered for pane X right now", the backend calls:

```ts
function renderedBuffer(id: SessionId): string[] {
  const term = getHeadless(id);
  const buf = term.buffer.active;
  const lines: string[] = [];
  for (let i = 0; i < buf.length; i++) {
    lines.push(buf.getLine(i)?.translateToString(true) ?? '');
  }
  return lines;
}
```

`translateToString(true)` trims trailing whitespace and gives us clean text per row. The rendered lines match what the browser's xterm.js renders for that pane.

## What carries over from ADR-003

- **The a11y-tree-in-browser idea is preserved** for frontend-driven tooling — xterm.js's `screenReaderMode: true` remains enabled on every `<Terminal />` (Phase 1). Future Phase 7+ features that need DOM-level a11y (for example, a browser-automation harness for the dashboard itself) still work.
- **The SSE event bus is unchanged.** Push-based events drive omniclaude's reactions as specified.
- **The batch-read contract is unchanged.** `readPane(id)` returns `{text_lines, cursor, cursor_style, last_render_ts}`. Just now the implementation is pure Node, not CDP.

## What changes from ADR-003

- **No Rust binary.** The "Fix: ship as user-installed prerequisite if distribution blocks MVP" fallback in Phase 3 becomes obsolete.
- **No sidecar process.** Headless reads are in-process Node function calls, sub-millisecond per pane.
- **Phase 3 gate**: `agent-browser batch read 10 panes ≤500ms` is now trivially satisfied (in-process array scan of ~200 lines × 10 panes = low-microseconds on any modern CPU). The gate target becomes more about the event-emission latency than read throughput.

## Consequences

**Positive**
- One less binary to distribute. Install stays `npm install -g theorchestra`.
- No cross-process RPC to debug.
- Deterministic read matches the browser's rendering exactly (same parser).
- Adds a small dep (`@xterm/headless` ~100KB) but no native build.

**Negative**
- Duplicated parsing cost — each session has two xterm parser instances (one in the browser, one in the backend). Memory overhead per pane: ~100-200KB for the headless terminal buffer. For 20 panes this is 2-4MB — trivial.
- Headless terminal resize needs to match the frontend's actual xterm dimensions. Otherwise rendered lines wrap differently between browser and server. We mitigate by routing the frontend's `resize` WS frame into `PtyManager.resize()` AND `headlessTerm.resize()` in the same call.

**Neutral**
- The "ship xterm-headless as an optional Phase 2+ feature" path doesn't exist anymore; it's core to Phase 3.

## Verification

- Phase 3 gate: feed a canned PTY stream into a `@xterm/headless` instance, assert the resulting `buffer.active` lines match a golden text file.
- Phase 3 gate: `renderedBuffer(id)` returns within 1ms per session in benchmarks.
- Phase 11 (new final phase): the "a11y read integrity" contract test becomes a xterm-headless round-trip with a canned ANSI stream.

## Migration note for Phase 7 omniclaude

Omniclaude's decision pipeline (ADR-003 §Correlation) is unaffected. When an SSE event arrives, omniclaude calls `backendClient.readOutput(id)` or a new `backendClient.renderedBuffer(id)` endpoint. The contract from omniclaude's POV is: "give me the text xterm renders." Whether the backend gets it via CDP on a Rust sidecar or via an in-process headless terminal is invisible to omniclaude.

## Update 2026-04-20 — `agent-browser` RE-ACCEPTED alongside `@xterm/headless`

This addendum originally framed `@xterm/headless` as a REPLACEMENT for
`agent-browser`. That framing was wrong — it conflated two different
observation contracts:

- `@xterm/headless` answers "what text would xterm have rendered for
  pane X right now" — byte-level VT state, per-pane, no DOM.
- `agent-browser` answers "what's the a11y tree of the DASHBOARD right
  now, with semantic element refs I can click/focus/type into" — DOM-
  level, whole-dashboard, AI-agent-ergonomic.

Both are needed. Per-pane status bar regex (Ctx%, permission prompts,
spinner glyphs) fits `@xterm/headless` cleanly. Orchestrator-level
observation of dashboard state + driving actions (clicking ✕ to kill
a pane, pressing Mode on a pane card, navigating tabs) fits
`agent-browser` cleanly.

Re-accepted as of REMEDIATION pivot 2026-04-20:

- `npm i -D agent-browser@0.26.0` — Rust CLI + persistent daemon, Chrome
  for Testing bundled via `agent-browser install`.
- `src/backend/agent-browser-observer.ts` — thin wrapper:
  `snapshotDashboard({dashboardUrl, session}) → {tree, latencyMs}` +
  `actOnRef(ref, 'click'|'hover'|'focus'|'dblclick', opts)`.
- `scripts/v3-agent-browser-gate.ts` — smoke gate; first live run
  returned the login-page snapshot in **223 ms** with 4 semantic refs
  and a successful hover round-trip. NFR-Perf-3 target (≤500 ms for
  10 panes) is in reach; full-load test deferred to Phase 10 dogfood.
- Companion pivot on frontend side: `@xterm/xterm` → `@wterm/react`
  (documented in `docs/adrs/v3.0-002-addendum-wterm-react.md`) —
  wterm's DOM + a11y-tree rendering is what makes `agent-browser`'s
  semantic locators land on meaningful targets.

Windows gotcha recorded in REMEDIATION.md + observer source: the
`node_modules/.bin/agent-browser.cmd` wrapper needs `shell: true` with
manual arg quoting when the repo path contains spaces ("Py Apps").
