# ADR-001: Adopt node-pty + xterm.js as the terminal substrate; drop WezTerm

- **Status**: Accepted (2026-04-19)
- **Supersedes**: v2.x WezTerm-mux-based substrate (documented in `docs/architecture.md`)
- **Context anchor**: `docs/v3.0-decisions.md` §1, §3

## Context

v2.x hosts every orchestrated session in a **WezTerm pane** and interacts with those panes by spawning `wezterm cli` subprocesses for every operation (read scrollback, send text, send keys, spawn pane, list panes). This substrate caused three cumulative failure classes:

1. **Socket saturation → GUI crashes.** Poll interval of 5s in `omni-watcher.cjs` saturated the WezTerm RPC socket and crashed the GUI; we had to back off to 30s (logged in `docs/architecture.md`). That 30s window is the ceiling for how fast omniclaude can react.
2. **Windows 10054 error class.** Cross-process `wezterm cli` calls through the WezTerm mux on Windows intermittently fail with `WSAECONNRESET` (10054). Every MCP tool has to guard for it, and the v2.x codebase accumulated retry wrappers around almost every call.
3. **Bifurcated UX.** Users interact with panes in the WezTerm GUI *and* in a browser dashboard at `:4200`. The two views drift (scrollback rendering, selection, copy-paste, theme). Remote/mobile access forces cloudflared-only-dashboard, which means GUI-only features stop working off-LAN.

v3.0 intentionally unifies the substrate so that **every pane is rendered in the dashboard**. The canonical substrate must therefore be a PTY backend that the dashboard process owns directly.

## Decision

We adopt **node-pty** (using ConPTY on Windows) as the PTY backend and **xterm.js** (via the `@wterm/react` wrapper) as the renderer. Both live in the same Node.js process as the dashboard server. The browser is a thin, reconnectable renderer; the source of truth for any pane is a PTY object in the backend, addressed by a durable `session_id`.

Key properties node-pty gives us:

- `pty.spawn(shell, args, { name, cols, rows, cwd, env })` returns a PTY object with `onData`, `write(data)`, `resize(cols, rows)`, and optional `handleFlowControl` for backpressure. No external binary, no RPC — just a Node handle.
- ConPTY on Windows handles escape-sequence translation natively; we inherit the stable terminal behaviour VS Code, Hyper, and Tabby ship on.
- Lifecycle is bound to the Node process, not to a GUI application. If the user closes the browser, the PTY keeps running.

WezTerm is dropped from the runtime dependency list. It remains installed on the user's machine only because v2.7.0 continues to run in parallel during migration (see ADR-004 / `v3.0-decisions.md` §7); once cutover completes, it can be uninstalled with no effect on v3.0.

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| **Keep WezTerm, fix socket throttling** | Does not address the bifurcated-UX problem. cloudflared-only dashboards still can't render WezTerm GUI. 10054 would still need retry wrappers. |
| **tmux / zellij as the mux, with a Rust CLI wrapper** | Buys us pane-persistence across dashboard restarts, but tmux on Windows needs WSL, and zellij has no mature ConPTY story. Adds a cross-process hop we wanted to remove. |
| **xterm.js directly, skip `@wterm/react`** | Workable, but the React integration surface (imperative `Terminal` API, manual fit/resize, manual addon loading) is boilerplate-heavy. `@wterm/react` encapsulates the lifecycle. |
| **Remote PTY server over SSH (fully split)** | Correct for multi-machine in Phase 2, overkill for single-user MVP. Deferred per `v3.0-decisions.md` §8. |
| **Full container-per-pane isolation (Docker)** | Massive install friction, no win for a single-user tool. |

## Consequences

**Positive:**
- One process owns every pane; no cross-process RPC for input/output. The 10054 error class disappears at the architectural level.
- Poll interval is no longer socket-limited — events can be push-based (SSE) at whatever cadence we want. This unlocks ADR-003 (active orchestrator).
- Phone/remote access parity: everything renders in the browser, so the mobile experience is identical to the desktop experience.
- node-pty is battle-tested (VS Code, Hyper, Tabby). We are not pioneering here.

**Negative / costs:**
- We lose WezTerm's multi-tab GUI — we must reimplement pane-layout, split-pane, and workspace concepts in the dashboard React app. MCP parity is achievable (`split_pane`, `list_workspaces`, `switch_workspace` all become frontend+backend state); GUI parity is not a goal.
- Dashboard-process crash takes all panes with it. Phase 2 can add a supervisor, but MVP accepts this (v2.7.0 runs in parallel as safety net — §7).
- `node-pty` requires a native build step. We will ship prebuilt binaries via `prebuild-install` to avoid requiring a C++ toolchain on every user's machine.

**Neutral:**
- Multi-LLM support is trivial: anything that runs as a TTY process (Claude, Codex, gemini-cli, droid, opencode, plain bash) spawns identically (`v3.0-decisions.md` §5).

## Verification

- Phase 1 exit gate: one Claude Code pane spawns via node-pty in the dashboard process, survives browser refresh, accepts `send_prompt` + `send_key` over MCP with no 10054 class error across 24h.
- Phase 2 exit gate: all 16 MCP tools implemented on the node-pty backend with identical signatures to v2.7.0.
