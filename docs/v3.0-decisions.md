# theorchestra v3.0 — Locked Decisions

Captured 2026-04-19 after structured scope-defining conversation. These are the architectural commitments that anchor the v3.0 spec and phased plan. Any change to any of these requires explicit user sign-off.

## Philosophy

Build the **most robust AI-orchestration substrate possible**, without restraint on time or scope. v3.0 replaces v2.7.0. v2.7.0 stays running in parallel on the user's machine until v3.0 is fully dogfooded and cut over.

---

## 1. Substrate — DROP WEZTERM ENTIRELY

- All terminals (regular shell AND AI-orchestration panes) live inside **wterm** panes rendered in the dashboard at `http://localhost:4200`.
- **No Windows Terminal bifurcation.** The whole point is full portability: user works at home, closes laptop, reopens on phone, keeps typing into the same pane. Windows Terminal/cmd/PowerShell/WSL are not used as shell hosts for v3.0 work.
- WezTerm is not required on the machine. Existing v2.7.0 installation stays during the transition, uninstalled once v3.0 is battle-tested.
- Kills the entire 10054 error class at the architectural level (no per-call `wezterm cli` spawns).

## 2. Omniclaude — ACTIVE ORCHESTRATOR

**Role change**: from passive router + event observer (v2.x) → active architect/orchestrator (v3.0).

**Autonomy split**:
- **Mechanics = autonomous** (decides without asking): "OK CONTINUE" prompts, auto-handoff when ctx fills, acking peer A2A messages, dispatching queued tasks per roadmap order, killing stuck panes, approving routine tool-use permission prompts.
- **Content = asks**: accept/reject a design decision, merge to main, feature direction choices, cross-project spawns, anything marked critical.

**Input corpus** (what omniclaude uses to decide):
- **agent-browser a11y-tree reads** against each pane's wterm rendering — deterministic, structured, no regex scrollback scraping.
- **Dashboard SSE events** — semantic events like `pane_idle`, `permission_prompt`, `peer_orphaned`, `ctx_threshold`, `a2a_received`. Push-based, real-time.
- Both correlated for robust decisions.

**User ↔ omniclaude chat**:
- Dashboard inline chat panel is primary.
- Telegram becomes **push-notification only**: "omniclaude has a question, open dashboard". No more terminal streaming to Telegram topics. Live streaming deprecated.
- Omniclaude conversation lives in dashboard; user never has to check two places.

## 3. Backend Stack

- **PTY backend**: Node.js + `node-pty` (ConPTY on Windows). Ships in the same process as the dashboard server. Matches existing wezbridge codebase language; mature library (VS Code, Hyper, Tabby use it).
- **UI framework**: `@wterm/react` + Vite. Port the current single-file 5000-line HTML into proper React components. No SSR (admin UI, doesn't need it).
- **Persistence**:
  - Close browser tab → pane survives (PTY keeps running in backend)
  - Restart dashboard → reconnect to running PTYs via durable session IDs
  - Reboot → auto-respawn Claude/Codex panes with `--continue` so sessions resume seamlessly
- **Install + launch**: `npm install -g theorchestra` + `theorchestra start` single command launches dashboard + PTY backend + (optional) Telegram integration together.

## 4. Auth + Remote Access

- Keep existing **cloudflared tunnel** for phone/remote access.
- Add a **bearer-token auth check** to the dashboard. Unknown requests → 401. Token saved in user's browser once per device. Protects against random cloudflared URL discovery.
- No OAuth, no multi-user, no user accounts. Single-user tool.

## 5. Multi-LLM Support

- Claude Code, Codex, gemini-cli, droid, opencode — **all first-class**. Any TUI CLI works as a pane. PTY backend doesn't care what process it spawns.
- A2A protocol works identically across LLMs because stdin is stdin. Codex already speaks the envelope format from v2.x.
- Persona system: Claude uses `~/.claude/agents/`; Codex uses its own agent config. Dashboard knows which persona source to pull from based on the CLI being spawned.

## 6. v2.x Carryover — ALL 12 FEATURES SURVIVE

### Core wezbridge protocol
1. **A2A envelope format + rules** — `[A2A from pane-X to pane-Y | corr=... | type=...]`, coordinator role declaration (parallel-worker / qa-verifier / pre-stager / monitor-only), push-vs-watch discipline, mandatory `send_key('enter')` after `send_prompt`.
2. **`/handoff` skill + structured handoff files** — 7-section template (Context / Current State / Open Threads / Next Steps / Constraints & Gotchas / Relevant Files / Corr ID), skill invocation pattern, external-write fallback for ctx-full panes.
3. **Auto-handoff daemon (Ctx-aware reset)** — 30% suggest / 50% enforce thresholds, READY/NOT_READY readiness-check protocol, 120s readiness poll, wait-for-idle-before-`/clear`, Ctrl+C + belt-and-suspenders enters, SSE broadcast.
4. **MCP tool surface (wezbridge namespace)** — `send_prompt`, `send_key`, `read_output`, `spawn_session`, `kill_session`, `auto_handoff`, `discover_sessions`, `split_pane`, `spawn_in_workspace`, `spawn_ssh_domain`, `set_tab_title`, `list_projects`, `list_workspaces`, `switch_workspace`, `wait_for_idle`, `get_status`. Same signatures, new node-pty-based implementation.

### v2.5 Agency Mode
5. **95+ persona pool** at `~/.claude/agents/` — spawn with `persona: 'reviewer' | 'coder' | 'frontend-design' | ...`. Dashboard shows persona badges.
6. **Git worktree isolation per pane** — `git worktree add <path> <branch>` for each pane that edits the same repo. Prevents parallel-agent merge conflicts.
7. **PRD-driven team bootstrap** — YAML file defines roles + tasks, dashboard spawns the whole team in parallel.
8. **`[PEER-PANE CONTEXT]` bootstrap prefix** — v2.7's `spawn_session spawned_by_pane_id` param auto-injects a header telling the spawned pane its own pane_id + coordinator pane_id + A2A reporting pattern. Bridges the persona-written-for-subagent-API gap.

### Memory + tracking + rules
9. **MemoryMaster integration (full stack)** — UserPromptSubmit recall hook, Stop auto-ingest every 15 messages, PreCompact warn-once hook (fixed 2026-04-19 — marker-based, stopped unconditional blocking), `mcp__memorymaster__*` tools on every pane. Cross-session memory is where "best practices" live — non-negotiable for omniclaude's decision quality.
10. **`active_tasks.md` task tracker protocol** — YAML frontmatter per task (status, owner, corr, created_at, dispatched_at, completed_at, stuck_threshold_min, follow_ups). Source of truth for "what's in flight"; omniclaude reads it to prioritize watching.
11. **Global CLAUDE.md rules + per-project overrides** — `~/.claude/CLAUDE.md` (A2A hard rules, coordinator role declaration, Intelligence-First Rule, Peer-Pane A2A Protocol section) + per-project CLAUDE.md files (repo architecture, stack, commands). Every spawned pane reads these at startup.
12. **Telegram project topic routing (bidirectional chat)** — one forum topic per project. Incoming topic-N text routes to pane-K; pane responses go back to topic-N. Bidirectional chat per pane from phone, even though live streaming is deprecated.

## 7. Migration Plan

**Parallel dual-run, gradual cutover.**

1. **v2.7.0 stays running** on wezterm on user's machine. Tag already pushed.
2. **v3.0 opens a new branch** (`v3.0`) off main. Development happens there in isolation.
3. **v3.0 spins up on a different port** (e.g. :4300 to avoid clashing with v2.7.0's :4200). Both can run side-by-side.
4. **Dogfood v3.0 on non-critical panes** for 1-2 weeks. Migrate pane-by-pane, confirm each workflow works.
5. **Once all panes are on v3.0**, stop v2.7.0, uninstall wezterm (or leave it — it doesn't hurt).
6. **Zero-downtime.** Reversible at any step.

## 8. Deferred for v3.0 Phase 2 (NOT in MVP)

- SSO via Cloudflare Access (token-auth covers MVP)
- Tailscale mesh alternative to cloudflared
- Multi-machine / remote PTY backends over SSH
- Full durable systemd-style supervisor (the node-pty-in-dashboard-process model is enough for MVP)
- Subagent support (`Task` tool integration — orthogonal)
