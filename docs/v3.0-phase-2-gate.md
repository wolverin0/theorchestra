## Deviations from Phase 2 plan

1. **`pane_id` field is now a string UUID** (v3.0 `SessionId`) instead of the numeric WezTerm pane_id used in v2.7. The field *name* is preserved for v2.7 signature compatibility; the *type* changed. External callers migrating from v2.7 MCP need to pass the string session_id they get from `discover_sessions`, not a number. Documented on every `pane_id` zod schema in the tool descriptions.

2. **`send_prompt` auto-appends `\r` at the backend**, not in the handler. v2.7's triple-redundancy pattern (two extra enters, one async) is obsolete — node-pty writes are reliable. Callers following the `CLAUDE.md` "always send_key('enter') after send_prompt" rule still work as belt-and-suspenders (extra enter on an empty input is a harmless no-op).

3. **Stubs for forward-phase tools**: `split_pane` returns `isError: true` (Phase 6 dependency); `spawn_ssh_domain` returns `isError: true` (deferred per `v3.0-decisions.md` §8); `auto_handoff` returns a non-error structured "pending_phase_4" payload with `planned_flow`; `switch_workspace` is a no-op with friendly text; `spawn_in_workspace` aliases `spawn_session`.

4. **`discover_sessions` `status` field is `'unknown'` in the list** to avoid N+1 backend calls. Real per-session status comes from `get_status`. This is a performance pragmatic, not a spec loss.

5. **B1 agent flagged a hidden bug**: the root `tsconfig.json` originally excluded `src/mcp/**`, so the handlers' typecheck was invisible to `v3:typecheck`. Fixed by adding `src/mcp/**/*.ts` to `include`. No typecheck failures once added.

6. **`import.meta.url` gotcha reappeared** in the MCP gate harness — package.json is `"type":"commonjs"` to preserve v2.7 `.cjs` scripts, so we use `__dirname` instead (per claim `mm-4200`).

7. **MCP SDK callback type-widening**: the SDK expects `CallToolResult` with an index signature, our `ToolResult` doesn't have one. Widened via a single cast at the register boundary in `src/mcp/server.ts` — handler-side typing stays clean, runtime validation still on the SDK.

## Follow-ups surfaced during Phase 2

1. The MCP SDK's `registerTool` callback typing is strict enough that our minimal `ToolResult` needs a widening cast. Not a blocker — just noted.
2. `discover_sessions`'s `status: 'unknown'` will be replaced with real per-session status once Phase 3 ships the SSE event bus.
3. `spawn_session` uses a 4-second blocking sleep before sending the initial prompt. Phase 3's `pane_idle` event makes this a proper wait-for-ready signal.
4. Backend now has 11 HTTP routes; if the list keeps growing Phase 3 should consider a router library (currently a hand-rolled if-chain). Not urgent.
